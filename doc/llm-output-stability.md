# LLM 출력 안정화/토큰 최적화 메모

소설 분석 파이프라인에서 실제로 겪은 이슈와 대응을 빠르게 참고하기 위한 운영 메모입니다.

---

## 1) 자주 보인 증상

- `summary`가 지나치게 짧거나(핵심 누락), 반대로 장황하게 길어짐
- `summary`/`events`에 유사 문장이 반복되어 중복 저장됨
- 캐릭터 `description`이 누적되며 중복 문구가 계속 붙음
- Groq `429 Too Many Requests`가 간헐적으로 발생

---

## 2) 왜 발생하나

### A. 프롬프트 제약 충돌/과제약

- Rules와 Expected JSON 예시의 문장 수 힌트가 다르면 모델이 더 강한 힌트(보통 예시)에 수렴
- `2-5 + 특정 글자 수 타겟` 같은 강한 길이 제약은
  - 분량을 억지로 맞추며 반복 문장을 만들거나
  - 반대로 핵심을 과도 압축하게 만들 수 있음

### B. LLM 출력 자체의 확률적 흔들림

- 같은 유형 입력에서도 문장 반복/표현 흔들림은 간헐적으로 발생 가능
- 그래서 프롬프트만으로 100% 방지는 어렵고, 저장 전후 후처리가 필요

### C. 토큰/요청량 압박

- RAG 컨텍스트, Known Characters 페이로드가 커지면 TPM 압박 증가
- 한 씬 처리 중 LLM 연속 호출이 있으면 RPM/TPM 모두 빠르게 소모

---

## 3) 현재 반영된 안정화 포인트

### A. summary 규칙 정리

- Rules 기준:
  - `summary`: `1-4` 문장, `\n` 결합
  - setup/change/result 중심으로 간결하게
  - 장문 원문 복붙/인용 금지(압축 패러프레이즈)
- Expected JSON의 `summary` 설명도 같은 방향으로 맞춤

### B. RAG 입력량 축소

- `scene_context_retriever.retrieve(..., top_k=2)`로 축소
- 파이프라인 호출부도 `top_k=2`로 고정

### C. 캐릭터 description 중복 누적 방지

- `CharacterRepository.upsert_character()`에서
  - 기존/신규 description을 구분자(`/`, `,`) 기준 파트 분해
  - 중복 파트 제거 후 재조합하여 저장

---

## 4) 운영 가이드

### summary가 너무 짧을 때

1. Rules와 Expected JSON의 힌트가 서로 일치하는지 먼저 확인
2. 문장 수 범위를 한 단계만 완화(예: `1-3` -> `1-4`)
3. 과한 글자수 타겟/하드 제한은 제거 또는 완화

### summary가 너무 길거나 반복될 때

1. 문장 수 상한을 소폭 축소(예: `1-6` -> `1-4`)
2. "장문 원문 복사 금지, 압축 패러프레이즈" 유지
3. 필요한 경우 저장 전 문장 dedupe 후처리 도입 검토

### 429가 자주 날 때

1. 먼저 RAG `top_k`와 Known Characters 페이로드 크기 축소
2. 연속 LLM 호출 구조(한 씬 내 다중 호출) 여부 점검
3. sleep은 보조 수단이며, 토큰/호출량 자체를 줄이는 조치가 우선

---

## 5) 핵심 원칙 (요약)

- 프롬프트만으로 완전 제어하려 하지 말고, 후처리와 함께 설계한다.
- 규칙(Rules)과 예시(Expected JSON)는 항상 같은 방향으로 맞춘다.
- 길이 제약은 "강한 하드 제한"보다 "완만한 범위"가 안정적이다.
- LLM 품질 문제와 rate-limit 문제를 분리해 진단한다.
